HTTP 的使用伴随开发周期的始终，今天我们有必要好好重新认识一下这位陌生的老朋友。

### 什么是 HTTP ？？？

我们打开维基百科可以看到下面的文字描述：

> 超文本传输协议（英语：HyperText Transfer Protocol，缩写：HTTP）是一种用于分布式、协作式和超媒体信息系统的应用层协议。

> HTTP 是万维网的数据通信的基础。设计 HTTP 最初的目的是为了提供一种发布和接收 HTML 页面的方法。通过 HTTP 或者 HTTPS 协议请求的资源由统一资源标识符（Uniform Resource Identifiers，URI）来标识。

原来在最初的时候，这个 HTTP 不是用来给我们每天调接口用的，人家和 HTML 是好哥俩。只不过随着互联网的技术的飞速发展，他也不断的迭代更新，变成了现在的模样。

那 HTTP 到底是什么模样？停下来想一下，我们对此还有印象吗？

整个互联网，小到一个公司，一个团队都会有自己的一套标准，这套标准在项目启动之初应该就定调了，不然对后面的研发工作将会非常不友好。正是因为这套请求配置的流程
在项目之初已经定调，后面我们开发的过程中基本都是依葫芦画瓢式使用。

依葫芦画瓢式的开发本身是没有错的，团队开发讲究效率和稳定，但是对我们开发个人，如果我们不去多想想背后发生了什么，是不利于我们个人成长的。

所以请继续向下看...

### HTTP 发送给服务器的数据放在哪里？

按数据放的位置来分的话，就下面三种

#### 放在 url 里面

严格遵循 HTTP 规范的 GET 请求都应该将发送给服务器的数据放在 url 中，可以是 url 的 path 或者 query 参数。

#### 放在 body 里面

严格遵循 HTTP 规范的 POST 请求都应该将发送给服务器的数据放在 body 里面

#### 放在 header 里面

还有数据会放在 header 里面？答案是肯定的。

首先 header 里面有着整个当前 HTTP 请求的信息，大概包含下面这些：

- Content-Length
- Content-Type
- Location
- User-Agent
- ...

我们也可以前后端约定好，放置我们对方都约定好的 key-value ，比如一些用来做权限校验的数据，当前登录用户的 token，等等。这个每个公司，每个项目
都不是一致规定的，这里非常的灵活。
